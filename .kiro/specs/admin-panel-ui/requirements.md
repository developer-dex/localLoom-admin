# Requirements Document

## Introduction

This document defines the requirements for the LocalLoom Admin Panel UI: a standalone single-page web application located at `/home/avesta/Documents/LocalLoom/localloom-admin` that consumes the LocalLoom backend admin APIs (mounted under `/api/admin`). The admin panel is built with TypeScript and React, uses shadcn/ui as the sole component library for primitive UI elements (button, input, table, dialog, etc.), integrates with the backend using a layered repository + API-client architecture, exposes a themeable design-token system so color schemes can be swapped in one place, and places every non-login route behind an authentication guard.

The LocalLoom backend currently exposes the following admin endpoints under `/api/admin`:
- `POST /auth/login`, `POST /auth/refresh-token`, `POST /auth/logout`, `GET /auth/profile`, `PATCH /auth/change-password`
- `GET|POST|PATCH|DELETE /categories` (create/update accept multipart with an `icon` image)
- `GET|POST|PATCH|DELETE /regions`

The admin panel MUST support these live endpoints on day one. The backend API surface documented in `localloom-backend/development-docs/api-list.md` additionally specifies upcoming admin modules (dashboard, users, tradies, locations-suburbs, reviews moderation, reports). The panel architecture MUST accommodate those modules through the same repository pattern without structural changes once their backend routes ship.

## Glossary

- **Admin_Panel**: The React single-page application hosted in `/home/avesta/Documents/LocalLoom/localloom-admin`.
- **Admin_Backend_Base_Url**: The base URL of the LocalLoom backend for admin calls, resolved from the `VITE_API_BASE_URL` environment variable at build time; the full admin prefix is `${VITE_API_BASE_URL}/api/admin`.
- **Admin_User**: A user with `role` equal to `admin` or `super_admin`, authenticated through the admin auth endpoints using email and password.
- **Access_Token**: A short-lived JWT returned by `POST /api/admin/auth/login` used in the `Authorization: Bearer` header.
- **Refresh_Token**: A long-lived JWT returned by `POST /api/admin/auth/login` used to obtain a new token pair via `POST /api/admin/auth/refresh-token`.
- **Auth_Storage**: The browser-side persistence layer (localStorage) that stores `accessToken`, `refreshToken`, and the cached admin profile.
- **Auth_Context**: The React context providing authentication state (current admin, login/logout functions, loading flag) to the rest of the app.
- **Protected_Route**: A route that renders its child only when a valid `accessToken` and admin profile are present in Auth_Context; otherwise it redirects to `/login`.
- **Api_Client**: The singleton HTTP client (axios instance) configured with `Admin_Backend_Base_Url`, request/response interceptors, and error normalization.
- **Repository**: A TypeScript class that exposes domain-focused methods for a single backend resource (for example `AdminCategoriesRepository`) and delegates HTTP work to Api_Client using values from Api_Paths.
- **Api_Paths**: A single TypeScript module `src/api/paths.ts` that exports path constants (for example `ADMIN_PATHS.categories.byId(id)`) used by every Repository.
- **Shadcn_Component**: A primitive UI component generated by the shadcn CLI into `src/components/ui/` (for example `src/components/ui/button.tsx`).
- **Shadcn_Wrapper**: A project-authored component in `src/components/wrappers/` that imports a Shadcn_Component and adds app-specific variants, defaults, or composition. Shadcn_Wrapper files MUST NOT be placed in `src/components/ui/`.
- **Design_Tokens**: CSS custom properties defined in `src/styles/tokens.css` that express brand colors, spacing, radii, and shadows, and are mapped to shadcn's CSS variables so changing tokens propagates across the entire admin panel.
- **Theme_Provider**: The React component in `src/providers/theme-provider.tsx` that applies the light or dark class to `document.documentElement` and persists the preference.
- **Steering_Rules_File**: The instruction file at `localloom-admin/.kiro/steering/admin-panel-rules.md` that records the mandatory standards every subsequent code change MUST follow.
- **Feature_Module**: A top-level folder under `src/features/<name>/` containing pages, hooks, repositories, and types for one admin domain (for example `src/features/categories/`).
- **App_Router**: The react-router configuration at `src/router/index.tsx` that wires public and Protected_Route routes.

## Requirements

### Requirement 1: Project Scaffolding and Tooling

**User Story:** As a developer, I want the admin panel to be bootstrapped with a standardized React + TypeScript + shadcn/ui stack so that every contributor starts from the same foundation.

#### Acceptance Criteria

1. THE Admin_Panel SHALL be generated with Vite using the `react-ts` template at the repository root `localloom-admin/`.
2. THE Admin_Panel SHALL declare `react`, `react-dom`, and `react-router-dom` version 6 or later as dependencies in `package.json`.
3. THE Admin_Panel SHALL declare `typescript` version 5 or later and `@types/react`, `@types/react-dom` as dev dependencies.
4. THE Admin_Panel SHALL declare `tailwindcss`, `postcss`, `autoprefixer`, `class-variance-authority`, `clsx`, `tailwind-merge`, and `lucide-react` as dependencies as required by shadcn/ui.
5. THE Admin_Panel SHALL declare `axios` as the HTTP client dependency.
6. THE Admin_Panel SHALL declare `@tanstack/react-query` version 5 or later for server-state management.
7. THE Admin_Panel SHALL declare `react-hook-form` and `zod` and `@hookform/resolvers` for form state and validation.
8. THE Admin_Panel SHALL configure path alias `@/*` mapped to `src/*` in both `tsconfig.json` and `vite.config.ts`.
9. THE Admin_Panel SHALL include a `components.json` file at the repository root configured for shadcn/ui with `style: "new-york"`, `baseColor: "slate"`, `cssVariables: true`, and aliases pointing to `@/components/ui`, `@/lib/utils`, `@/hooks`.
10. THE Admin_Panel SHALL configure ESLint with `@typescript-eslint` rules, and Prettier, and SHALL declare an `npm run lint`, `npm run typecheck`, and `npm run build` script that each exit with a non-zero status on failure.
11. THE Admin_Panel SHALL declare a `.env.example` file at the repository root containing `VITE_API_BASE_URL=http://localhost:5000` and `VITE_APP_NAME=LocalLoom Admin`.
12. THE Admin_Panel SHALL commit a `.gitignore` that excludes `node_modules`, `dist`, `.env`, and editor-local files.

### Requirement 2: shadcn/ui Adoption and Wrapping Strategy

**User Story:** As a developer, I want shadcn/ui to be the exclusive source of primitive UI components so that I never hand-roll a button, input, or table and can safely re-run the shadcn CLI without losing customizations.

#### Acceptance Criteria

1. THE Admin_Panel SHALL install primitive UI components (for example button, input, label, card, table, dialog, sheet, dropdown-menu, select, toast, form, tabs, badge, avatar, separator, skeleton, tooltip, pagination, alert, alert-dialog) using the shadcn CLI into `src/components/ui/`.
2. THE Admin_Panel SHALL NOT include any project-authored component file inside `src/components/ui/` other than files produced by the shadcn CLI.
3. WHEN an application-specific variant or default prop is required for a Shadcn_Component, THE developer SHALL create a new Shadcn_Wrapper file under `src/components/wrappers/<component-name>.tsx` that imports the original Shadcn_Component and re-exports an extended variant.
4. THE Shadcn_Wrapper files SHALL NOT modify the contents of files in `src/components/ui/`.
5. WHEN a feature needs a UI primitive (button, input, table, dialog, etc.), THE Feature_Module SHALL import that primitive either directly from `@/components/ui/<name>` or through a Shadcn_Wrapper at `@/components/wrappers/<name>`.
6. IF a file under `src/components/ui/` is edited by hand, THEN the Steering_Rules_File SHALL flag this as a violation that must be reverted and replaced with a Shadcn_Wrapper.
7. THE Admin_Panel SHALL expose the shadcn `cn` helper from `@/lib/utils.ts` and SHALL use it for conditional class merging in every component.

### Requirement 3: Theming and Design Tokens

**User Story:** As a brand owner, I want a single source of truth for colors and design tokens so that swapping the shadcn color scheme updates the entire admin panel without touching individual components.

#### Acceptance Criteria

1. THE Admin_Panel SHALL define all color tokens as CSS custom properties in `src/styles/tokens.css` using the HSL triplet format expected by shadcn (for example `--primary: 222.2 47.4% 11.2%;`).
2. THE Admin_Panel SHALL define both `:root` (light) and `.dark` (dark) blocks in `src/styles/tokens.css` providing values for every token consumed by shadcn components: `background`, `foreground`, `card`, `card-foreground`, `popover`, `popover-foreground`, `primary`, `primary-foreground`, `secondary`, `secondary-foreground`, `muted`, `muted-foreground`, `accent`, `accent-foreground`, `destructive`, `destructive-foreground`, `border`, `input`, `ring`, and `radius`.
3. THE `tailwind.config.ts` file SHALL map Tailwind color names (`primary`, `secondary`, `destructive`, `muted`, `accent`, `card`, `popover`, `border`, `input`, `ring`, `background`, `foreground`) to `hsl(var(--token))` expressions so that changing a token in `tokens.css` propagates through every Tailwind class usage.
4. THE Admin_Panel SHALL import `src/styles/tokens.css` exactly once from `src/main.tsx` before any component stylesheet is imported.
5. THE Theme_Provider SHALL read the user preference from `localStorage` under the key `localloom-admin-theme` with allowed values `light`, `dark`, or `system`, SHALL apply the resolved mode by toggling the `dark` class on `document.documentElement`, and SHALL persist the user's selection back to `localStorage` whenever it changes.
6. THE Admin_Panel SHALL render a theme toggle control in the main layout header that switches between `light`, `dark`, and `system` modes via the Theme_Provider.
7. THE Admin_Panel SHALL NOT hardcode hex or named color values (for example `#ff0000`, `text-red-500`) in feature components; WHERE a brand color is needed, the component SHALL reference a token class such as `bg-primary`, `text-destructive`, or `border-border`.

### Requirement 4: API Integration Architecture

**User Story:** As a developer, I want a structured API layer with a single HTTP client, centralized path constants, and per-resource repositories so that every backend call follows the same pattern and error handling.

#### Acceptance Criteria

1. THE Admin_Panel SHALL expose a single Api_Client instance at `src/api/client.ts` created with `axios.create({ baseURL: \`${import.meta.env.VITE_API_BASE_URL}/api/admin\` })`.
2. THE Api_Client SHALL attach a request interceptor that reads the current Access_Token from Auth_Storage and sets the `Authorization: Bearer <token>` header on every outgoing request when a token exists.
3. WHEN a response with HTTP status 401 is received and a Refresh_Token is available in Auth_Storage, THE Api_Client SHALL call `POST /auth/refresh-token` exactly once per original request, store the new token pair, and retry the original request with the new Access_Token.
4. IF the refresh attempt in criterion 3 fails with any status other than 200, THEN THE Api_Client SHALL clear Auth_Storage, dispatch a logout event to Auth_Context, and reject the original request.
5. THE Api_Client SHALL normalize non-2xx responses into an `ApiError` object exposing `status: number`, `message: string`, and `errors?: string[]`, sourcing `message` and `errors` from the backend envelope `{ success, message, errors? }`.
6. THE Admin_Panel SHALL declare all backend paths as constants in `src/api/paths.ts` under a single `ADMIN_PATHS` object (for example `ADMIN_PATHS.auth.login = "/auth/login"`, `ADMIN_PATHS.categories.root = "/categories"`, `ADMIN_PATHS.categories.byId = (id: string) => \`/categories/${id}\``).
7. THE Feature_Module code SHALL NOT embed raw path strings; every Repository SHALL construct URLs using values from `ADMIN_PATHS`.
8. THE Admin_Panel SHALL implement one Repository class per backend resource under `src/features/<name>/<name>.repository.ts` exposing async methods that return domain types and throw `ApiError` on failure.
9. THE Admin_Panel SHALL wrap every Repository call in a React Query `useQuery` or `useMutation` hook under `src/features/<name>/hooks/` so that caching, loading, and error states are handled uniformly.
10. WHEN a mutation modifies data that another query has cached, THE mutation hook SHALL invalidate the affected query keys using `queryClient.invalidateQueries`.
11. THE Admin_Panel SHALL render error messages from `ApiError.message` via a toast using the shadcn toast primitive; WHERE `ApiError.errors` is non-empty, THE toast SHALL display the first entry as a secondary line.
12. WHEN a Repository method uploads a file (for example category icon), THE Repository SHALL accept a `FormData` payload, set `Content-Type: multipart/form-data`, and pass the payload through Api_Client unchanged.

### Requirement 5: Admin Authentication Flow

**User Story:** As an admin, I want to sign in with my email and password and remain signed in across page reloads so that I can access the admin panel without re-entering credentials every time.

#### Acceptance Criteria

1. WHEN the `/login` route is visited, THE Admin_Panel SHALL render a login form with `email` and `password` fields validated by zod (email format, password minimum length 1 for input, required).
2. WHEN the login form is submitted, THE Admin_Panel SHALL POST `{ email, password }` to `POST /api/admin/auth/login` via the auth Repository.
3. WHEN the login response returns HTTP 200 with `data.tokens.accessToken`, `data.tokens.refreshToken`, and `data.admin`, THE Auth_Context SHALL store `accessToken` and `refreshToken` in Auth_Storage under keys `localloom-admin-access-token` and `localloom-admin-refresh-token`, store the admin profile under `localloom-admin-profile`, and navigate the user to `/dashboard`.
4. IF the login response returns HTTP 401, THEN THE Admin_Panel SHALL display a toast with the message from the error envelope and SHALL NOT navigate away from `/login`.
5. IF the login response returns HTTP 403, THEN THE Admin_Panel SHALL display a toast with the backend message (for example `"Admin account is not active"`) and SHALL NOT navigate away from `/login`.
6. WHEN the Admin_Panel mounts and an Access_Token exists in Auth_Storage, THE Auth_Context SHALL call `GET /api/admin/auth/profile` once to hydrate the current admin; IF the call succeeds, THEN THE Auth_Context SHALL mark the user as authenticated; IF the call fails with HTTP 401, THEN THE Auth_Context SHALL clear Auth_Storage.
7. WHEN the user triggers logout from the layout user menu, THE Auth_Context SHALL POST `POST /api/admin/auth/logout`, clear Auth_Storage, and navigate to `/login` regardless of the logout response status.
8. WHEN a Protected_Route is visited and Auth_Context reports the user as unauthenticated, THE App_Router SHALL redirect to `/login` and preserve the requested path in a `redirectTo` query parameter.
9. WHEN a Protected_Route is visited and Auth_Context is still initializing (profile hydration in progress), THE App_Router SHALL render a full-screen loading skeleton using the shadcn skeleton primitive.
10. WHEN the user visits `/login` while already authenticated, THE App_Router SHALL redirect to `/dashboard`.
11. THE Admin_Panel SHALL provide a `/change-password` route rendering a form that calls `PATCH /api/admin/auth/change-password` with `currentPassword` and `newPassword`; on success the Admin_Panel SHALL display a toast and clear the form.

**User Story:** As an admin, I want a consistent layout with a sidebar, top bar, and content area so that I can navigate modules and see who I am signed in as.

#### Acceptance Criteria

5. THE current route SHALL be visually indicated in the sidebar by highlighting the active entry using a token-driven `bg-accent` background.
6. THE layout SHALL wrap its content area with the shadcn `Toaster` component so that toasts rendered by any feature appear globally.

### Requirement 8: Category Management Module

**User Story:** As an admin, I want to list, create, edit, and soft-delete service categories so that I can curate the catalog tradies choose from.

#### Acceptance Criteria

1. WHEN the user visits `/categories`, THE category page SHALL call `GET /api/admin/categories` and render the results in a shadcn `Table` with columns Name, Icon, Sort Order, Active, Created At, and Actions.
2. THE category table SHALL support client-side filtering by name through a shadcn `Input` search box above the table.
3. WHEN the user clicks the New Category button, THE Admin_Panel SHALL open a shadcn `Dialog` containing a form with fields `name` (required, max 100 chars), `description` (optional, max 500 chars), `sortOrder` (optional, integer, min 0), and `icon` (optional image file).
4. WHEN the Create form is submitted, THE Admin_Panel SHALL POST `multipart/form-data` to `POST /api/admin/categories` via the `AdminCategoriesRepository` and, on success, close the dialog, display a success toast, and invalidate the categories list query.
5. WHEN the user clicks Edit on a category row, THE Admin_Panel SHALL open the same dialog pre-filled with the category's current values and, on submit, PATCH `multipart/form-data` to `PATCH /api/admin/categories/:id`.
6. WHEN the user clicks Delete on a category row, THE Admin_Panel SHALL open a shadcn `AlertDialog` asking for confirmation and, on confirm, DELETE `/api/admin/categories/:id`.
7. WHEN the backend returns HTTP 400 with validation errors on any category mutation, THE form SHALL display the first error beside the relevant field if the field name is included in the error payload, and in a toast otherwise.
8. IF a category create or update fails with HTTP 409 (duplicate name), THEN THE form SHALL display the backend message under the `name` field.

### Requirement 11: Protected Routing and Access Control

**User Story:** As a security-conscious product owner, I want every admin-panel page other than the login page to require authentication so that the panel cannot be browsed by unauthenticated users.

#### Acceptance Criteria

1. THE App_Router SHALL declare exactly one public route at path `/login`.
2. THE App_Router SHALL wrap every non-login route (including `/dashboard`, `/categories`, `/regions`, `/change-password`, and any `404`) with a Protected_Route component.
3. WHEN Protected_Route mounts and Auth_Context reports no Access_Token, THE Protected_Route SHALL navigate to `/login?redirectTo=<current path>`.
4. WHEN the user successfully logs in and a `redirectTo` query parameter is present, THE login page SHALL navigate to the value of `redirectTo` on success, provided the value starts with `/` and does not contain `://`.
5. IF a request to any `/api/admin/*` endpoint returns HTTP 401 after a failed refresh, THEN THE Api_Client SHALL clear Auth_Storage and the Admin_Panel SHALL navigate to `/login`.
6. THE App_Router SHALL render a 404 page for unknown paths, which SHALL itself be wrapped by Protected_Route.

### Requirement 14: Environment Configuration and Build

**User Story:** As a developer, I want build-time environment variables for the backend base URL so that the same code runs against local, staging, and production backends.

#### Acceptance Criteria

1. THE Admin_Panel SHALL read `VITE_API_BASE_URL` at build time and SHALL fail the build with a descriptive error (via a typed env module at `src/config/env.ts`) when the value is empty or does not start with `http://` or `https://`.
2. THE Admin_Panel SHALL expose a typed env module at `src/config/env.ts` exporting `{ apiBaseUrl: string, appName: string }` parsed from `import.meta.env` and validated by zod.
3. THE Admin_Panel SHALL NOT reference `import.meta.env` directly outside `src/config/env.ts`.
4. THE `npm run build` command SHALL produce a static bundle under `dist/` that can be served from any static host, and SHALL NOT require a Node.js runtime to serve the built output.
